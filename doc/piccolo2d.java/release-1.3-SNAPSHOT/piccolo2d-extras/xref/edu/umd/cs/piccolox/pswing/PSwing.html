32  
33    We are currently developing Piccolo, a "scenegraph" for use in 2D graphics.
34    One of our ultimate goals is to support Swing lightweight components
35    within Piccolo, whose graphical space supports arbitray affine transforms.
36    The challenge in this pursuit is getting the components to respond and
37    render properly though not actually displayed in a standard Java component
38    hierarchy.
39  
40  
41    The first issues involved making the Swing components focusable and
42    showing.  This was accomplished by adding the Swing components to a 0x0
43    JComponent which was in turn added to our main Piccolo application component.
44    To our good fortune, a Java component is showing merely if it and its
45    ancestors are showing and not based on whether it is ACTUALLY visible.
46    Likewise, focus in a JComponent depends merely on the component's
47    containing window having focus.
48  
49  
50    The second issue involved capturing the repaint calls on a Swing
51    component.  Normally, for a repaint and the consequent call to
52    paintImmediately, a Swing component obtains the Graphics object necessary
53    to render itself through the Java component heirarchy.  However, for Piccolo
54    we would like the component to render using a Graphics object that Piccolo
55    may have arbitrarily transformed in some way.  By capturing in the
56    RepaintManager the repaint calls made on our special Swing components, we
57    are able to redirect the repaint requests through the Piccolo architecture to
58    put the Graphics in its proper context.  Unfortunately, this means that
59    if the Swing component contains other Swing components, then any repaint
60    requests made by one of these nested components must go through
61    the Piccolo architecture then through the top level Swing component
62    down to the nested Swing component.  This normally doesn't cause a
63    problem.  However, if calling paint on one of these nested
64    children causes a call to repaint then an infinite loop ensues.  This does
65    in fact happen in the Swing components that use cell renderers.  Before
66    the cell renderer is painted, it is invalidated and consequently
67    repainted.  We solved this problem by putting a lock on repaint calls for
68    a component while that component is painting.  (A similar problem faced
69    the Swing team over this same issue.  They solved it by inserting a
70    CellRendererPane to capture the renderer's invalidate calls.)
71  
72  
73    Another issue arose over the forwarding of mouse events to the Swing
74    components.  Since our Swing components are not actually displayed on
75    screen in the standard manner, we must manually dispatch any MouseEvents
76    we want the component to receive.  Hence, we needed to find the deepest
77    visible component at a particular location that accepts MouseEvents.
78    Finding the deepest visible component at a point was achieved with the
79    "findComponentAt" method in java.awt.Container.  With the
80    "getListeners(Class listenerType)" method added in JDK1.3 Beta we are able
81    to determine if the component has any Mouse Listeners. However, we haven't
82    yet found a way to determine if MouseEvents have been specifically enabled
83    for a component. The package private method "eventEnabled" in
84    java.awt.Component does exactly what we want but is, of course,
85    inaccessible.  In order to dispatch events correctly we would need a
86    public accessor to the method "boolean eventEnabled(AWTEvent)" in
87    java.awt.Component.
88  
89  
90    Still another issue involves the management of cursors when the mouse is
91    over a Swing component in our application.  To the Java mechanisms, the
92    mouse never appears to enter the bounds of the Swing components since they
93    are contained by a 0x0 JComponent.  Hence, we must manually change the
94    cursor when the mouse enters one of the Swing components in our
95    application. This generally works but becomes a problem if the Swing
96    component's cursor changes while we are over that Swing component (for
97    instance, if you resize a Table Column).  In order to manage cursors
98    properly, we would need setCursor to fire property change events.
99  
100 
101   With the above fixes, most Swing components work.  The only Swing
102   components that are definitely broken are ToolTips and those that rely on
103   JPopupMenu. In order to implement ToolTips properly, we would need to have
104   a method in ToolTipManager that allows us to set the current manager, as
105   is possible with RepaintManager.  In order to implement JPopupMenu, we
106   will likely need to reimplement JPopupMenu to function in Piccolo with
107   a transformed Graphics and to insert itself in the proper place in the
108   Piccolo scenegraph.
109 
110 */

113 /**
114  * <b>PSwing</b> is used to add Swing Components to a Piccolo canvas.
115  * <p>
116  * Example: adding a swing JButton to a PCanvas:
117  * <pre>
118  *     PSwingCanvas canvas = new PSwingCanvas();
119  *     JButton button = new JButton("Button");
120  *     swing = new PSwing(canvas, button);
121  *     canvas.getLayer().addChild(swing);
122  * <pre>
123  * </p>
124  * <p>
125  * NOTE: PSwing has the current limitation that it does not listen for
126  *       Container events.  This is only an issue if you create a PSwing
127  *       and later add Swing components to the PSwing's component hierarchy
128  *       that do not have double buffering turned off or have a smaller font
129  *       size than the minimum font size of the original PSwing's component
130  *       hierarchy.
131  * </p>
132  * <p>
133  *       For instance, the following bit of code will give unexpected
134  *       results:
135  *       <pre>
136  *            JPanel panel = new JPanel();
137  *            PSwing swing = new PSwing(panel);
138  *            JPanel newChild = new JPanel();
139  *            newChild.setDoubleBuffered(true);
140  *            panel.add(newChild);
141  *       </pre>
142  * </p>
143  * <p>
144  * NOTE: PSwing cannot be correctly interacted with through multiple cameras.
145  * There is no support for it yet.
146  * </p>

148  * NOTE: PSwing is java.io.Serializable.
149  * </p>

270             component.revalidate();
271         }
272 
273         if( shouldRenderGreek( renderContext ) ) {
274             paintAsGreek( g2 );
275         }
276         else {

319     /**
320      * Renders to a buffered image, then draws that image to the
321      * drawing surface associated with g2 (usually the screen).
322      *
323      * @param g2 graphics context for rendering the JComponent
324      */
325     public void paint( Graphics2D g2 ) {
326         if( component.getBounds().isEmpty() ) {
327             // The component has not been initialized yet.
328             return;
329         }
330 
331         PSwingRepaintManager manager = (PSwingRepaintManager)RepaintManager.currentManager( component );
332         manager.lockRepaint( component );
333 
334         Graphics2D bufferedGraphics = null;
335         if( !isBufferValid() ) {
336             // Get the graphics context associated with a new buffered image.
337             // Use TYPE_INT_ARGB_PRE so that transparent components look good on Windows.
338             buffer = new BufferedImage( component.getWidth(), component.getHeight(), BufferedImage.TYPE_INT_ARGB_PRE );
339             bufferedGraphics = buffer.createGraphics();
340         }
341         else {
342             // Use the graphics context associated with the existing buffered image
343             bufferedGraphics = buffer.createGraphics();
344             // Clear the buffered image to prevent artifacts on Macintosh
345             bufferedGraphics.setBackground( BUFFER_BACKGROUND_COLOR );
346             bufferedGraphics.clearRect( 0, 0, component.getWidth(), component.getHeight() );
347         }
348 
349         // Start with the rendering hints from the provided graphics context
350         bufferedGraphics.setRenderingHints( g2.getRenderingHints() );
351 
352         //PSwing sometimes causes JComponent text to render with "..." when fractional font metrics are enabled.  These are now always disabled for the offscreen buffer.
353         bufferedGraphics.setRenderingHint( RenderingHints.KEY_FRACTIONALMETRICS, RenderingHints.VALUE_FRACTIONALMETRICS_OFF );
354 
355         // Draw the component to the buffer
356         component.paint( bufferedGraphics );
357 
358         // Draw the buffer to g2's associated drawing surface
359         g2.drawRenderedImage( buffer, IDENTITY_TRANSFORM );
360 
361         manager.unlockRepaint( component );
362     }

411     /**
412      * We need to turn off double buffering of Swing components within
413      * Piccolo since all components contained within a native container
414      * use the same buffer for double buffering.  With normal Swing
415      * widgets this is fine, but for Swing components within Piccolo this
416      * causes problems.  This function recurses the component tree
417      * rooted at c, and turns off any double buffering in use.  It also
418      * updates the minimum font size based on the font size of c and adds
419      * a property change listener to listen for changes to the font.
420      *
421      * @param c The Component to be recursively unDoubleBuffered
422      */
423     void init( Component c ) {
424         Component[] children = null;
425         if( c instanceof Container ) {
426             children = ( (Container)c ).getComponents();
427         }
428 
429         if( c.getFont() != null ) {
430             minFontSize = Math.min( minFontSize, c.getFont().getSize() );
431         }
432 
433         if( children != null ) {
434             for( int j = 0; j < children.length; j++ ) {
435                 init( children[j] );
436             }
437         }
438 
439         if( c instanceof JComponent ) {
440             ( (JComponent)c ).setDoubleBuffered( false );
441             c.addPropertyChangeListener( "font", this );
442             c.addComponentListener( new ComponentAdapter() {
443                 public void componentResized( ComponentEvent e ) {
444                     computeBounds();
445                 }
446 
447                 public void componentShown( ComponentEvent e ) {
448                     computeBounds();
449                 }
450             } );
451         }
452     }

472     /**
473      * Attaches a listener to the specified node and all its parents to listen
474      * for a change in the PSwingCanvas.  Only PROPERTY_PARENT listeners are added
475      * so this code wouldn't handle if a PLayer were viewed by a different PCamera
476      * since that constitutes a child change.
477      * @param node The child node at which to begin a parent-based traversal for adding listeners.
478      */
479     private void listenForCanvas( PNode node ) {
480         //need to get the full tree for this node
481         PNode p = node;
482         while( p != null ) {
483             listenToNode( p );
484 
485             PNode parent = p;
486 //            System.out.println( "parent = " + parent.getClass() );
487             if( parent instanceof PLayer ) {
488                 PLayer player = (PLayer)parent;
489 //                System.out.println( "Found player: with " + player.getCameraCount() + " cameras" );
490                 for( int i = 0; i < player.getCameraCount(); i++ ) {
491                     PCamera cam = player.getCamera( i );
492                     if( cam.getComponent() instanceof PSwingCanvas ) {
493                         updateCanvas( (PSwingCanvas)cam.getComponent() );
494                         break;
495                     }
496                 }
497             }
498             p = p.getParent();
499         }
500     }

502     /**
503      * Attach a listener to the specified node, if one has not already been attached.
504      * @param node the node to listen to for parent/pcamera/pcanvas changes
505      */
506     private void listenToNode( PNode node ) {
507 //        System.out.println( "listeningTo.size() = " + listeningTo.size() );
508         if( !listeningTo( node ) ) {
509             listeningTo.add( node );
510             node.addPropertyChangeListener( PNode.PROPERTY_PARENT, parentListener );
511         }
512     }

514     /**
515      * Determine whether this PSwing is already listening to the specified node for camera/canvas changes.
516      * @param node the node to check
517      * @return true if this PSwing is already listening to the specified node for camera/canvas changes
518      */
519     private boolean listeningTo( PNode node ) {
520         for( int i = 0; i < listeningTo.size(); i++ ) {
521             PNode pNode = (PNode)listeningTo.get( i );
522             if( pNode == node ) {
523                 return true;
524             }
525         }
526         return false;
527     }

529     /**
530      * Removes this PSwing from previous PSwingCanvas (if any), and ensure that this PSwing is attached to the new PSwingCanvas.
531      * @param newCanvas the new PSwingCanvas (may be null)
532      */
533     private void updateCanvas( PSwingCanvas newCanvas ) {
534         if( newCanvas != canvas ) {
535             if( canvas != null ) {
536                 canvas.removePSwing( this );
537             }
538             if( newCanvas != null ) {
539                 canvas = newCanvas;
540                 canvas.addPSwing( this );
541                 reshape();
542                 repaint();
543                 canvas.invalidate();
544                 canvas.revalidate();
545                 canvas.repaint();
546             }
547         }
